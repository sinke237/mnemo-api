Product Overview

Mnemo is a personal productivity and learning application that merges two distinct but complementary experiences: spaced-repetition flashcard study (powered by the Mnemo API's SM-2 engine) and Android-native digital wellness (real-time device usage tracking, per-app limits, AI-powered overuse insights, and Focus Mode). The app targets a single authenticated user, is distributed on Google Play and App Store, and uses a clean 4-tab bottom navigation structure.

Three User Personas Driving Design

PersonaPrimary PainKey Features They Use
Kofi, 24 — Daily Learner Distracted by social media; forgets to review decks Study Sessions Streak Focus Mode Limits
Amara, 30 — Content Creator Tedious card creation; hard to spot weak areas CSV Import Deck Mgmt Weak Spots Plans
Lena, 19 — Wellness-Focused Unaware of app time; no enforcement mechanism Usage Dashboard App Limits AI Insights

Core Design Principles

🎯
Calm productivity aesthetic. The seed color #4A90D9 (calm blue) is chosen deliberately for learning contexts — scientifically associated with focus and reduced anxiety. Dark surfaces are used for study screens to minimize eye strain during long sessions.
♿
Accessibility-first. All touch targets ≥ 48×48dp, color never the sole state indicator, semantic labels on all interactive elements, text scales with system preferences.
🤖
Platform-aware degradation. Android-exclusive features (usage tracking, hard limits, Focus Mode) are visible on iOS with clear "Not available on iOS" labels — never hidden, never confusing.

05

Study Plan & Alert System

⏰ Full-Screen Study Alert System

When a scheduled study time arrives, Mnemo delivers a full-screen, interruptive experience — functionally identical to an OS alarm — using Android's full-screen notification intent + SCHEDULE_EXACT_ALARM / AlarmManager. The signature sound is consistent and memorable: users learn to associate it with "it's study time."

PHASE 01
Plan Creation
User sets up plan via Settings → Study Plan. Selects deck, duration, daily minutes, daily_minutes, and an alert time (time picker). AlarmManager / WorkManager schedules a daily exact alarm at that time.
PHASE 02
Alarm Fires
At the exact scheduled time, Android fires a PendingIntent with FLAG_IMMUTABLE + EXACT. A BroadcastReceiver receives it and launches the Alert Activity with showOnLockScreen and turnScreenOn flags.
PHASE 03
Full-Screen Intent
A high-priority notification with fullScreenIntent launches StudyAlertActivity. The notification channel has IMPORTANCE_HIGH + bypassDnd = true (user-confirmable). Takes over the entire screen.
PHASE 04
Sound Plays
A distinct, recognizable melody plays via MediaPlayer or RingtoneManager. Sound continues looping until user interacts. User can select from 3 preset sounds + custom ringtone picker.
PHASE 05
User Responds
Study Now: stop sound → launch Session Config → Active Session.
Snooze: stop sound → reschedule alarm (default 10 min, configurable 5/10/15/30 min) → return to last screen.
Skip: stop sound → dismiss → mark session missed.
PHASE 06
Missed Sessions
If no response for 2 minutes, sound stops but notification persists in tray. Home screen shows "Missed today's session" chip in amber. Plan progress marks the day incomplete (not failed — plan continues).

Study Plan Scheduling Rules

ScenarioBehavior
User creates planAlarmManager schedules one exact alarm per day at the configured alert_time, for days 1–N of the plan. Each alarm carries plan_id, deck_id, day_num.
User changes alert timeAll pending alarms cancelled (by plan_id hashCode); re-scheduled with new time.
User cancels planAll alarms for that plan_id cancelled.
Alarm fires while studyingIf user is actively in a session, alarm is suppressed (session state checked). Alarm logged as "completed early."
Device rebootedBootCompletedReceiver re-registers all future alarms from the active plan stored in Drift DB.
Do Not Disturb activeStudy alert channel configured with setBypassDnd(true) — always rings (user must explicitly opt out in channel settings). Notification warns user this bypasses DND.
App force-closedAlarmManager survives app force-close (it's an OS service). StudyAlarmReceiver is launched even if app is dead.
Snooze limitNo hard snooze limit enforced (unlike app-limit overlay). User can snooze indefinitely. Missed sessions are informational only.

Signature Study Alert Sound

🎵
The Mnemo study sound must be instantly recognizable. It is consistent across ALL study alerts so users develop a Pavlovian association: hear this sound → time to study. It should be:

• Melodic and pleasant (not alarming/anxious) — learning should feel positive
• Distinct from system sounds (custom asset bundled in assets/sounds/mnemo_alert.mp3)
• 3–5 second loop, loops until interaction
• Pairs with a distinct 3-beat haptic pattern (long-short-long)

Edge Cases & System Behavior

Authentication Edge Cases

⚠️ Token Expiry Mid-Session
JWT expires while user is answering a card
AuthInterceptor catches 401 on answer submission. Silently re-exchanges api_key + user_id for new JWT (user never sees interruption). Retries original answer request once. Answer is not lost.
⚠️ Invalid API Key
Regex validated before submit but user pastes wrong format
Client-side regex rejects before any network call. Inline error: "Key must match format: mnm_live_[64 hex chars]". Submit button disabled.
⚠️ Network Unreachable at Startup
App launched with no internet connection
Full-screen error state: "No connection · Mnemo requires internet to sync. Check your connection and retry." Retry button. App does not loop or crash.
⚠️ Refresh Returns 401
api_key revoked server-side
Clear all credentials from FlutterSecureStorage. Redirect to Onboarding. Show toast: "Your credentials were revoked. Please re-authenticate."

Session Edge Cases

ℹ️ Session Already Ended
User submits answer after session times out concurrently
Server returns 409 SESSION_ALREADY_ENDED. App dismisses session screen, shows snackbar "Session already ended", auto-fetches and shows summary.
ℹ️ No Cards Available
Deck has no eligible cards for selected mode
Server returns 409 NO_CARDS_AVAILABLE. Dialog: "No eligible cards in this deck for this session type. Try removing filters or adding new cards." Two CTAs: Change Settings / View Deck.
⚠️ Network Drop Mid-Session
User loses connection while answering
NetworkError caught by Dio. Persistent banner at bottom: "No connection · Your last answer wasn't saved. Retry?" Retry button attempts resubmission. Timer pauses visually (but still counts on server).
⚠️ Focus Mode Permission Revoked
User revokes DND access from Android settings mid-session
BroadcastReceiver detects NOTIFICATION_POLICY_ACCESS_SETTINGS_CHANGED. Focus Mode toggle automatically flips to OFF. Snackbar: "Focus Mode disabled — DND permission was revoked."

Wellness / Usage Edge Cases

ℹ️ Uninstalled App
App with a limit configured is uninstalled by user
Drift record persists. App Limit screen shows "(app removed)" suffix. Record auto-pruned after 30 days. UsageMonitorService skips uninstalled packages gracefully.
ℹ️ 1000+ Apps Installed
Device with many apps; UsageStats returns large dataset
Dashboard shows top 30 by usage time only. "See all" opens a searchable, alphabetically sorted full list. Drift stores full dataset; processing happens off main thread.
⚠️ Service Killed (Device Reboot)
UsageMonitorService killed after device restart
BootCompletedReceiver re-starts foreground service via startForegroundService. Study alarms are also re-registered here from Drift DB plan data.
⚠️ Snooze Limit Maxed (3×)
User has snooze 3 times for the app-limit overlay
"5 More Minutes" button is disabled (grayed out with lock icon). Only "I'm Done" remains. Tooltip on disabled button: "Snooze limit reached for today."

Import Edge Cases

⚠️ File Too Large
User selects CSV > 5MB
App reads file size after file picker returns path. If > 5MB, shows dialog: "File too large (X.X MB). Maximum is 5 MB. Please split your CSV and import in parts." No upload attempted.
⚠️ Non-UTF-8 File
CSV saved with Latin-1 or Windows-1252 encoding
Server returns 400 INVALID_CSV_FORMAT. Error dialog: "Invalid encoding. Re-save your CSV as UTF-8 (e.g., in Excel: Save As → CSV UTF-8)." Help icon links to documentation.
ℹ️ Import Job Stuck
Job stays in "processing" for > 5 minutes
Polling times out at 300s. Message: "Import taking longer than expected. You'll be notified when complete — feel free to keep using Mnemo." User can leave the screen.
ℹ️ Both deck_id and deck_name
Bug: both fields somehow provided
Client-side validation prevents this. If existing deck selected from dropdown, deck_name field is disabled. If "New deck" toggle active, deck dropdown is disabled.

UX Rationale

Why Dark Mode as Default?

Study sessions typically happen in the evening or at night (after school/work). Bright white interfaces cause eye strain and disrupt sleep patterns due to blue light exposure. The dark-first design also reduces battery consumption on OLED screens — important for extended study sessions on mobile devices.

Why 4-Tab Bottom Navigation?

The 4-tab structure maps cleanly to the app's four conceptual domains: What do I study now? (Home), What are my materials? (Decks), How am I doing? (Progress), and Am I using my phone well? (Wellness). Five tabs would crowd the bar; three would require hiding distinct features. Tab icons are self-explanatory with labels always visible for clarity.

Why Full-Screen Alert (Not Just a Notification)?

Standard notifications are trivial to dismiss without reading. The spec's goal is behavioral change — building a study habit. Full-screen interruptive alerts (identical in mechanism to alarm apps and incoming calls) create the required urgency and saliency. The signature sound builds conditioned association over time. The "Start Now / Snooze / Skip" decision forces conscious choice rather than passive dismissal.

Why the Timer Chip is Non-Intrusive by Default?

A large countdown clock creates anxiety that degrades recall performance. The compact chip form (small pill in top-right corner) allows users who find timers helpful to reference it without it dominating the cognitive space of the card question. The expand-on-tap behavior accommodates both user types.

Why Separate Decks and Home?

Home is a curated action surface — it shows what needs attention now (due cards, weak spots, plan reminder). Decks is a management surface — it's where you browse, organize, import. Users in "review mode" mindset don't want to wade through deck management; users in "organization mode" don't need due-count urgency.

Why Shimmer Skeletons vs Spinners?

Skeleton screens communicate what content is coming — they reduce perceived loading time by 20–30% (per Facebook's Inertial UX research). Spinners provide no shape context. Since Mnemo has well-defined content shapes (deck cards, stat grids), skeletons are always the right choice for initial loads.

Why Platform Features Are Visible (But Labeled) on iOS?

Hiding Android-only features on iOS could confuse users who switch devices or read documentation. Showing them with clear "Not available on iOS" labels educates users about the platform difference and avoids "where did that feature go?" confusion. It also provides a natural upgrade prompt for Android.

Why SM-2 Scores Are Abstracted from the User?

The raw SM-2 score (0–5) is an internal API detail. Users see their answer evaluated as correct/incorrect with a confidence selector (Low/Medium/High = 1/2/3). The confidence feeds into the score calculation server-side. Exposing raw 0–5 scores would require users to learn the algorithm — unnecessary cognitive overhead that adds friction to each answer.

Why No Offline Mode in MVP?

Offline mode requires careful conflict resolution, sync queues, and significantly more engineering investment. The spec correctly defers it to P1 roadmap. The MVP makes every network error explicit and actionable (retry buttons, not silent failures) so users understand the app's online dependency without confusion.

Why Exact Alarms Instead of Inexact?

Study sessions are time-sensitive — a 30-minute drift in alarm delivery (common with inexact alarms due to battery optimization) would break user trust in the system. setExactAndAllowWhileIdle ensures the alarm fires within seconds of the scheduled time, even in Doze mode. The permission (SCHEDULE_EXACT_ALARM on Android 12+) must be requested with clear rationale.
